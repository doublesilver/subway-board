테이블: daily_visits

CREATE TABLE daily_visits (
  id SERIAL PRIMARY KEY,
  visit_date DATE NOT NULL,
  subway_line_id INTEGER,
  visit_count INTEGER DEFAULT 0

CREATE TABLE visit_logs (
  id SERIAL PRIMARY KEY,
  visitor_hash VARCHAR(16),
  subway_line_id INTEGER,
  visited_at TIMESTAMP DEFAULT NOW(),
  referrer VARCHAR(255),
  device_type VARCHAR(20)
);

신규 테이블: hourly_visits

CREATE TABLE hourly_visits (
  id SERIAL PRIMARY KEY,
  visit_date DATE NOT NULL,
  visit_hour SMALLINT NOT NULL, -- 7, 8
  subway_line_id INTEGER,

옵션 3: 고유 방문자 테이블 (DAU/WAU/MAU)

옵션 3 안정성 추천

신규 테이블: unique_visitors

CREATE TABLE unique_visitors (
  id SERIAL PRIMARY KEY,
  visitor_hash VARCHAR(16) NOT NULL,
  visit_date DATE NOT NULL,
  first_line_id INTEGER,
  lines_visited SMALLINT DEFAULT 1,
  UNIQUE(visitor_hash, visit_date)
);

추천 조합

A. 가벼운 분석

daily_visits + hourly_visits

  • 용량 부담 최소 (~21KB/월)
  • 시간대별 피크 분석 가능
  • 구현 간단

B. 중간 분석

daily_visits + unique_visitors

  • DAU/WAU/MAU 측정
  • 재방문율 분석
  • 서버 재시작에도 안정적

C. 풀 분석

daily_visits + visit_logs + unique_visitors

  • 모든 분석 가능
  • 가장 상세한 인사이트
  • 마케팅/사업 분석에 활용

선택 가이드

"일단 가볍게 시작하고 싶어" 옵션 2 (시간대별)
"재방문율이 궁금해" 옵션 3 (UV)
"서버 재시작해도 안정적이었으면" 옵션 3 (UV)
"나중에 마케팅 분석도 할거야" 옵션 1 + 3 (풀 분석)
